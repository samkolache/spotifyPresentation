Application Changes Communications

Login Name Changes

DocuSign CLM has redefined login name criteria. DocuSign CLM login names can now begin with either an alpha or numeric character only. Login names can no longer begin with special characters (e.g., ! ? & , etc.).

Workflow Changes

DocuSign will now automatically cancel any workflow that has been running for two (2) years or more.

Deprecation: Public eForms

As of November 2020, DocuSign CLM eForms cannot be made public. If your account has at least one active public eForm, you can still use this functionality without interruption. eForms can still contain CSS and attributes but, from November 2020 onward, use of JavaScript in new forms is not supported.

  • Communicate to specific users about new login name criteria
  • Communicate to users about new workflow changes
  • Communicate to users about changes to public eForms and the use of Javascript within eForms.

  • We can encrypt student’s SSN to make the data harder to access

Scenario #2

Sample Email To Stakeholders to Discuss Possible Alternatives

Dear [Stakeholder Name],

I hope this email finds you well. I wanted to follow up on your suggestion to auto-publish SSNs with full names and grades on our secure website. As a team, we have discussed this proposal and have some concerns about the potential risks associated with this approach. We believe that student privacy must be protected at all times, and publishing SSNs alongside student names and grades could compromise this privacy. Additionally, there may be legal or regulatory implications to consider, depending on where our school or university is located. Finally, we want to ensure that our students are comfortable with the way their information is being shared, and we are concerned that some may not be comfortable having their SSN published online.

As an alternative, we suggest using the Student ID as a unique identifier to connect grades with each student. The Student ID can be used as a primary key and is a unique identifier that does not compromise student privacy. If stakeholders are concerned about the security of the Student ID, we can suggest hashing or encrypting the ID to ensure that it remains secure.

We cannot stress enough the importance of protecting Personally Identifiable Information (PII) at all times. This is a classified data level 1 and must be safeguarded against any unauthorized access, disclosure, or use. If this information is compromised, it can result in significant legal trouble, reputation damage, and identity theft.

Finally, we want to remind stakeholders of the privacy rights that students have. Any release of student information must be done with the student's permission, and their privacy rights must be respected at all times.

Thank you for your understanding and your commitment to our students' well-being.

Scenario #3

SaaS Migration Plan

The Spotify Legal department has decided to move to a new Legal Invoicing platform (SaaS cloud based). You (and your team) will be the administrators for this platform. Put together a plan of activities you would want to complete to ensure a successful implementation of the application. Your old Legal invoicing platform is also cloud based. You should consider data, user access and existing workflows

  1. Define Scope and Objective
  2. Conduct Data Inventory and Migration
  3. User Access Management and Design Workflows
  4. Testing and Integration
  5. Rollout, Monitor, and Provide Support

Scenario #3

Define Scope and Objective

  • Define the project scope, which includes goals, deadlines, and deliverables
  • Engage in discussions with stakeholders to ensure that our planning is aligned appropriately.
  • Identify any problems early on
  • Create a project scope statement

Scenario #3

Conduct Data Inventory and Migration

  • Identify possible data to be transferred, invoices, user profiles, etc.
  • Perform data mapping with this data
  • Create a backup of the current data
  • Find out the most efficient way to migrate the data
  • Conduct a trial migration
  • Execute the migration and verify the data was transferred accurately

Scenario #3

User Access Management and Defining Workflows

  • Create use cases for all workflows in the new platform
  • Identify all users and give permissions based on use cases
  • Identify existing workflows, invoicing, billing, collection, etc
  • Configure workflows on the new platform to match workflows on the old one
  • Find out how existing workflows can be optimized using the new platform

Scenario #3

Testing and Integration

  • Test platform for functionality, compatibility, and security
  • Conduct a quality assurance test to find any bugs or errors
  • Check to see if things like the accounting and billing systems are integrated
  • Set up UAT to make sure workflows are configured correctly
  • Validate once more if all data has been transferred correctly

Scenario #3

Rollout, Monitor, and Provide Support

  • Launch the platform in a controlled manner
  • Monitor the performance while traffic starts to build within the platform
  • Provide support by creating training documents and other resources for returning users
  • Evaluate any user feedback and take action if needed
